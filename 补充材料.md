## 1. 实际数据划分与来源（对应报告 3.2 的更正）
- **唯一来源**：`test-time-compute/game-of-24`（1362 道可解题，按 `Rank` 难度排序）。训练/测试**同源切分**，代码只 `load_dataset("test-time-compute/game-of-24")`，**全程未加载 nlile/24-game**。
- **train**：Rank 1–1262（最易，solved_rate 高），1262 道 → 同时用其求解轨迹做 SFT。
- **hard-test（OOD-hard）**：Rank 1263–1362（最难 100，人类 solved_rate 20.7%–58%）。
- **幻觉集**：合成 100 个**无解**四元组（`{1..13}⁴` 减去可解集；`solve24()` 验证 100 题全无解）。**不是** nlile solvable=False（nlile 已无该字段）。
- **Countdown OOD**：`Jiayi-Pan/Countdown-Tasks-3to4` test 共 1024，**评测实际取前 256 条**（`eval_compare --countdown_n 256`）。
- **无泄漏**：train(1262) ∩ test(100) = ∅，且均与幻觉集不相交（数据准备与评测两处均有机械断言）。

## 2. 实际运行环境（对应报告 3.1 的更正）
- 框架 **TinyZero（= veRL）** + FSDP；**rollout 后端 = HF rollout**（`actor_rollout_ref.rollout.name=hf`），**非 vLLM**——本环境 vLLM ABI 损坏（`undefined symbol: cuTensorMapEncodeTiled`，CUDA cu118 与 cu12.x 不匹配）。
- 为此对 veRL 的改动：
  - `verl/workers/rollout/hf_rollout.py`：加 `num_return_sequences=n`，使 HF rollout 支持 GRPO 组内多采样（否则 n=4 退化为 1）。
  - `verl/utils/torch_functional.py`：加 `VERL_DISABLE_FLASH_ATTN_CE` 开关，绕开 flash-attn 交叉熵的 Triton 崩溃。
  - `verl/trainer/main_ppo.py`：把上述 env 经 ray `runtime_env` 传播到 worker。
  - `verl/trainer/ppo/ray_trainer.py`：加 `final_val` 开关。
- 硬件：**8×A100-80GB**，bf16；多 arm 用**隔离 Ray 集群**（独立端口/temp-dir，每 arm 2–4 GPU）并行。

## 6. 分数口径与统计说明（务必区分"准确率"与"平均奖励"）

### 6.0 准确率 ≠ 平均奖励（报告未说清，最易误读）
- **准确率 / 求解率（accuracy / solved rate）**：一道题**真正答对**——算式恰好用给定 4 个数各一次、且结果 **= 24**（即 `reward == 1.0`）。
  评测里的 **`greedy_acc` / `best_verified@k` / `n_correct`** 都是这个口径（`score >= 1.0` 的题目占比；而 `score==1.0` 仅当真正答对才给）。代码：`scripts/eval_compare.py` L104-105、`examples/eval_ood.py`；奖励：`verl/utils/reward_score/game24.py` L118。**这是本报告所有"X%"准确率的统一定义，不是把格式分也算进去的"默认 accuracy"。**
- **平均奖励（mean reward）**：每题 reward 的平均，**含 0.1 格式分**（shaped 模式还含"接近 24"的部分分）。训练时的 `val/test_score`（`ray_trainer.py` L440 = `np.mean(rewards)`）与表里的 `Test mean` 都是这个。**它不是准确率。**
  - 例：难测试 `mean = 0.109` ≈ 1 题对(1.0) + 99 题仅格式(0.1) → **准确率 = 1%**；
  - 例：4.2 中 shaped 的 `val/test_score = 0.54` **绝不是 54% 准确率**，而是大量"接近 24"部分分堆出来的平均奖励，与 sparse 的 0.109 **不可直接比较**。
- **换算关系（sparse）**：`mean ≈ acc×1.0 + (有格式但错)×0.1 + (无答案)×0`。报告凡出现 0.x 的"得分/score"，请标注是"平均奖励"，避免被读成准确率。

### 6.1 统计噪声
- **hard-test n=100 → 1% = 1 道题**。表中 0%↔2% 的差异在统计噪声内，**难测试贪心**不足以单独支撑"谁更好"；更可靠的判据是 **best-of-N（同 100 题多采样，方差小）/ in-dist(100) / Countdown(256)**。
- **幻觉指标的局限**：幻觉集为无解题，结构上**不存在**用给定 4 数等于 24 的算式，故"求解率（score==1.0）"对任何模型（含 base）**恒为 0%**。因此"0% 编造率"是预期结果，不能单独作为"模型不瞎编"的强证据；严格的幻觉测试应衡量"是否自信给出声称=24 但实则不等于的算式"或"是否会判定无解"。

## 7. 实验命名约定（报告缺）
- **Phase-1**：base（纯 GRPO）vs SFT→GRPO 的受控对比（基于 SFT **v1**）。
- **Wave-1**：基于 SFT **v2** 的奖励/采样消融（control / n8 / shaped / staged-p1）。
- **Div\***：多样性/熵坍缩消融（扫 KL / ent / T）。
- **sftv2base**：SFT v2 未做 GRPO 的起点。
- **bv@k** = best-of-N-verified@k；**maj@k** = 多数投票表达式是否正确；**greedy** = 1-shot 确定性解码。

## 8. 最终交付与结论（报告缺"单一明确陈述"）
- **交付模型**：SFT v2（冷启动）；如需提升分布内贪心，可在其上做短程 GRPO（control，~80 步）。
- **难测试集成绩**：贪心 ≈ 0–3%（接近 Qwen2.5-1.5B 单次解码上限）；**可部署的 best-of-N-verified：SFT v2 bv@16 ≈ 21%、bv@64 = 50%（T=1.5）**。
- **分布内**：SFT v2 = 27%（贪心）；+GRPO(control@80) = 42%。
- **OOD（Countdown 256）**：SFT→GRPO(v1) = 12.89%（纯 GRPO 2.34%、SFT-only 5.08%）。
- **关键发现**：① SFT 冷启动是 GRPO 在稀疏奖励下有效的前提（纯 GRPO 训练奖励长期 ~0.1）；② GRPO 抬高 greedy/in-dist 但**压低 best-of-N/多样性**（熵坍缩，需早停或 KL 锚定缓解）；③ 测试时计算（best-of-N + 高温采样）可释放至 50% 的潜力。

## 10.2 SFT 两版（合规，共享同 1262 训练题）

## 10.4 四种 self-critic prompt 设计（⑤ critic 轴；评测方法，非训练消融）
| 设计 | 脚本 | 关键 | 结果(精确率) |
